# 무신사(MUSINSA) 웹 E2E 테스트 자동화 프레임워크
![Python](https://img.shields.io/badge/Python-3776AB?style=flat-square\&logo=python\&logoColor=white)
![Selenium](https://img.shields.io/badge/Selenium-43B02A?style=flat-square\&logo=selenium\&logoColor=white)
![Test](https://img.shields.io/badge/Test-E2E%20Automation-FF5722?style=flat-square)

무신사(MUSINSA) 웹 서비스의 핵심 사용자 흐름을 기반으로 **E2E UI 테스트 자동화 프레임워크를 설계 및 구현한 프로젝트**이다.

단순 기능 확인을 넘어, 실제 사용자 관점 문제를 사전 탐지하고 **품질 보증 중심 테스트 전략과 자동화 구조를 구축**하는 데 목적을 두었다.

<br>

## 1\. 프로젝트 개요

  * **진행 기간** : 2026.02 \~ 2026.03
  * **테스트 대상** : [무신사 웹 페이지](https://www.musinsa.com)
  * **사용 기술** : Python, Selenium WebDriver
  * **테스트 유형** : E2E 테스트, 기능 테스트, 예외 테스트, 데이터 검증 테스트

<br>

## 2\. 목표 및 테스트 전략

본 프로젝트는 단순 기능 동작 여부를 확인하는 수준을 넘어, **실제 사용자 행동 흐름 기반 테스트 설계**에 초점을 맞췄다.

  * #### 사용자 흐름 기반 E2E 테스트

    상품 검색 → 필터 적용 → 정렬 → 상세 페이지 → 장바구니까지 이어지는 **실제 구매 과정 전체 검증**

  * #### 데이터 정합성 검증

    UI 표시 여부가 아닌, 할인율 계산, 가격 정렬 등 **데이터 정확성 검증**

  * #### 예외 시나리오 테스트

    빈 검색어 입력, 존재하지 않는 키워드 검색 → **실제 서비스에서 자주 발생하는 예외 상황 대응 검증**

  * #### 자동화 및 재사용 구조 설계

    반복 테스트 최소화, 동적 웹 환경에서도 안정적인 실행, 테스트 선택 실행 기능 구현

<br>

## 3\. 테스트 케이스

총 **12개의 핵심 시나리오**를 기반으로 테스트를 구성했다.

| TC ID         | 구분  | 테스트 명      | 검증 내용              |
| :------------ | :---- | :------------- | :--------------------- |
| TC\_SEARCH\_01  | 기능  | 정상 검색      | 검색 결과가 1개 이상 노출    |
| TC\_SEARCH\_02  | 예외  | 존재하지 않는 검색 | 결과 없음 메시지 또는 상품 0개 |
| TC\_SEARCH\_03  | 예외  | 빈칸 검색      | 경고 알림 표시           |
| TC\_FILTER\_01  | 기능  | 남성 필터 적용   | 남성/공용 상품만 노출       |
| TC\_FILTER\_02  | 검증  | 여성 상품 배제   | 여성 상품 미노출          |
| TC\_FILTER\_03  | 기능  | 복수 필터 적용   | 성별 + 색상 조건 만족      |
| TC\_SORTING\_01 | 기능  | 낮은 가격순 정렬     | 가격 오름차순            |
| TC\_SORTING\_02 | 기능  | 할인율순 정렬      | 할인율 내림차순           |
| TC\_PRODUCT\_01 | UI/UX | 상세 페이지 이동  | 정상 페이지 로딩          |
| TC\_PRODUCT\_02 | 검증  | 할인율 계산     | 계산값 = 표시값          |
| TC\_CART\_01    | 기능  | 장바구니 담기    | 상품 정상 추가           |
| TC\_CART\_02    | UI/UX | 장바구니 수량 확인    | 수량 증가 확인           |

<br>

## 4\. 테스트 시나리오 동작 흐름

앞서 정의한 테스트 케이스가 실제 브라우저 환경에서 **어떤 흐름으로 실행되고, 어떤 기준으로 검증되는지**를 명확히 기술했다.

  * #### TC_SEARCH_01 (정상 검색)

      * **동작** : 메인 페이지에 진입한 후 검색창에 특정 키워드(예: "후드티")를 입력하여 검색을 수행한다.
      * **검증** : 화면에 로딩된 상품 리스트의 요소를 수집하여 상품 개수를 확인한다.
      * **결과** : 상품이 1개 이상 정상적으로 노출되면 `PASS`, 요소를 찾지 못하거나 상품이 존재하지 않으면 `FAIL`로 판정한다.

  * #### TC_SEARCH_02 (존재하지 않는 검색)

      * **동작** : 검색창에 무작위 난수 키워드(예: test_123456)를 생성한 후 검색을 수행한다.
      * **검증** : 명시적 대기를 활용하여 "결과 없음" 안내 문구의 노출 여부를 확인하거나, 상품 리스트의 개수가 0인지 확인한다.
      * **결과** : 안내 문구가 정상적으로 노출되거나 상품이 존재하지 않으면 `PASS`, 상품이 노출되는 경우 `FAIL`로 판정한다.

  * #### TC_SEARCH_03 (빈칸 검색)

      * **동작** : 검색창에 공백(" ")을 입력한 후 검색을 수행한다.
      * **검증** : 시스템 알림창(Alert)의 발생 여부와 메시지를 확인한다.
      * **결과** : 알림창이 정상적으로 노출되면 `PASS`, 알림 없이 페이지가 이동하면 `FAIL`로 판정한다.

  * #### TC_FILTER_01 (남성 필터 적용)

      * **동작** : 특정 키워드(예: "맨투맨")로 검색을 수행한 후 남성 필터를 적용하고, 스크롤을 통해 상품을 추가로 로딩한다.
      * **검증** : 로딩된 상품 중 최대 30개를 무작위로 추출하여 각 상품의 성별 태그가 '남성' 또는 '공용'인지 확인한다.
      * **결과** : 모든 상품이 조건을 만족하면 `PASS`, 조건을 만족하지 않거나 요소를 찾을 수 없으면 `FAIL`을 누적하여 판정한다.

  * #### TC_FILTER_02 (여성 상품 배제)

      * **동작** : 특정 키워드(예: "맨투맨")로 검색을 수행한 후 남성 필터를 적용하고, 스크롤을 통해 상품을 추가로 로딩한다.
      * **검증** : 무작위로 추출한 최대 30개의 상품을 기준으로 '여성' 태그가 포함되어 있는지 확인한다.
      * **결과** : '여성' 태그가 하나라도 존재하면 `FAIL`, 전부 배제되면 `PASS`로 판정한다.

  * #### TC_FILTER_03 (복수 필터 적용)

      * **동작** : 성별(남성)과 색상(레드) 필터를 동시에 적용한 후 상품 목록을 로딩한다.
      * **검증** : 무작위로 추출한 최대 30개의 상품에 대해 성별이 '남성' 또는 '공용'인지, 색상이 '레드'인지 각각 확인한다.
      * **결과** : 두 조건을 모두 만족하면 `PASS`, 조건을 만족하지 않거나 요소를 찾을 수 없으면 `FAIL`로 판정한다.

  * #### TC_SORTING_01 (낮은 가격순 정렬)

      * **동작** : 특정 키워드(예: "후드티")로 검색을 수행한 후 정렬 기준을 낮은 가격순으로 변경하고, 스크롤을 통해 상품을 로딩한다.
      * **검증** : 상위 15개 상품의 가격을 추출하여 콤마와 '원'을 제거한 뒤 정수로 변환하고, 오름차순으로 정렬되어 있는지 확인한다.
      * **결과** : 이전 값이 다음 값보다 작거나 같으면 `PASS`, 더 낮은 가격이 뒤에 존재하면 `FAIL`로 판정한다.

  * #### TC_SORTING_02 (할인율순 정렬)

      * **동작** : 검색 결과에서 정렬 기준을 할인율순으로 변경한 후 상품 목록을 로딩한다.
      * **검증** : 상위 15개의 할인율을 추출하여 '%'를 제거한 뒤 정수로 변환하고, 내림차순으로 정렬되어 있는지 확인한다.
      * **결과** : 이전 값이 다음 값보다 크거나 같으면 `PASS`, 더 높은 할인율이 뒤에 존재하거나 값이 누락된 경우 `FAIL`로 판정한다.

  * #### TC_PRODUCT_01 (상품 상세 페이지 이동)

      * **동작** : 검색 결과 목록에서 상품을 로딩한 후 임의의 상품 1개를 선택하여 클릭하고 상세 페이지로 이동을 수행한다.
      * **검증** : 클릭한 상품의 상세 페이지로 정상적으로 전환되었는지 확인한다.
      * **결과** : 새 탭이 정상적으로 열리고 페이지 전환이 성공하면 `PASS`, 탭이 생성되지 않거나 페이지 진입에 실패하면 `FAIL`로 판정한다.

  * #### TC_PRODUCT_02 (할인율 계산)

      * **동작** : 검색 결과에서 무작위로 5개의 상품을 선택하여 각각의 상세 페이지에 진입한 후 데이터 수집을 수행한다.
      * **검증** : 원가, 할인율, 판매가를 추출한 뒤 `원가 * (100 - 할인율) / 100` 수식으로 계산한 값과 실제 판매가를 비교하여 일치 여부를 확인한다.
      * **결과** : 계산값과 실제 값이 일치하면 `PASS`, 오차가 발생하거나 데이터 추출에 실패하면 `FAIL`을 누적하여 판정하고 다음 상품으로 진행한다.

  * #### TC_CART_01 (장바구니 담기)

      * **동작** : 로그인한 후 상품 상세 페이지에 진입하여 임의의 상품을 선택하고 장바구니 담기를 수행한다.
      * **검증** : 장바구니 페이지로 이동한 뒤, 상세 페이지에서 확인한 상품명이 장바구니 목록에 존재하는지 확인한다.
      * **결과** : 상품이 정상적으로 존재하면 `PASS`, 존재하지 않으면 `FAIL`로 판정하며, 모든 옵션이 품절된 경우 `SKIP`으로 처리한다.

  * #### TC_CART_02 (장바구니 수량 확인)

      * **동작** : 상품을 담기 전 장바구니 수량을 확인한 후 장바구니 담기를 수행하고, 이후 변경된 수량을 다시 확인한다.
      * **검증** : 상품 추가 전후의 수량을 비교하여 값이 증가했는지 확인한다.
      * **결과** : 수량이 정상적으로 증가하면 `PASS`, 변화가 없거나 오류가 발생하면 `FAIL`로 판정한다.

## 5\. 주요 구현 포인트

본 프로젝트는 테스트 자동화 구현뿐만 아니라, **실행 구조와 테스트 안정성을 고려한 설계**에 중점을 두었다.

  * #### main.py

    테스트 선택 실행 기능, 사용자 입력 기반 실행 구조, 드라이버를 필요할 때만 불러오도록 적용

  * #### 명시적 대기

    명시적 대기를 활용하여 동적 요소 로딩을 안정적으로 처리하고, 테스트 실패 가능성을 최소화

  * #### 테스트 책임 분리

    main.py → 실행 및 제어 / test 코드 → 검증 수행 → 역할 분리를 통해 유지보수성과 확장성 향상

  * #### 자동화 구조 설계

    반복 테스트 최소화, 기능별 테스트 모듈 분리, 재사용 가능한 함수 기반 구조 설계

<br>

## 6\. 프로젝트 구조

페이지 동작과 테스트 로직을 분리하여 **유지보수성과 확장성**을 고려한 구조로 설계했다.

```text
MUSINSA_TEST
├── pages/
│   └── musinsa_page.py      # 페이지 요소 및 공통 동작
├── tests/
│   ├── search/              # 검색 테스트
│   ├── filter/              # 필터 테스트
│   ├── sorting/             # 정렬 테스트
│   ├── product/             # 상품 테스트
│   └── cart/                # 장바구니 테스트
├── utils/
│   └── driver.py            # WebDriver 설정
└── main.py                  # 테스트 실행
```

<br>

## 7\. 테스트 실행 방법

`main.py`를 실행하면 테스트 목록이 출력되며, 원하는 테스트를 선택하여 실행할 수 있다.

  * #### 실행 예시

<!-- end list -->

```text
0 | 전체 테스트
1 | TC_SEARCH_01
2 | TC_SEARCH_02
...

번호 입력 (예: 1 3 5, 종료 = n)
```

  * `0` 입력 → 전체 테스트 실행
  * 숫자 입력 → 선택 테스트 실행
  * `n` 입력 → 종료
